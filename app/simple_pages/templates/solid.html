The solid principles improve the reliability of our code by working on its structure and its logical consistency.

The SOLID principles are:

  • The Single-Responsibility Principle (SRP)
  • The Open-Closed Principle (OCP)
  • The Liskov Substitution Principle (LSP)
  • The Interface Segregation Principle (ISP)
  • The Dependency inversion Principle (DIP)


Single-Responsibility Principle

It means a class should have only one reason to change.

There are subclasses Addition(), Subtraction(), Multiplication(), Division() of base class Calculator as shown in image() which demonstrates Single-Responsibility Principle.


image(a) Calculator program demonstrates Inheritance


For instance, Division class has one function called divide() on line 52 of image(a). This function returns the result of that function. So it has only one task to return a result. Thus, if we change anything in this function, it would not affect on entire software behaviour. It will only affect this class method.

In this above image of the calculator program has four methods namely add, subtract, multiply, division.

Open-Closed Principle

It means that the software entities should be open for extension but closed for modification.

As discussed the earlier the subclasses Addition(), Subtraction(), Multiplication(), Division() inherit the property of class Calculation.


image(b) Calculations_test.py file of calculator program


As shown in image(b) of calculator program, we can see the subclasses inherited the tuple list data from calculation class. Now, they are extending the new process as described in the image (For instance, line number 38,39 and 40).

Liskov Substitution Principle

It means that Functions that use pointers or references to base classes must be able to use objects of derived classes without knowing it.


image(c) Calculator program demonstrates Encapsulation


According to the Inheritance principle, when subclasses Addition(), Subtraction(), Multiplication(), Division() inherit the property of base classes, we would not be able to change parent class. We have to re-write it. But in Liskov Substitution principle, we can duplicate the parent class in subclasses and able to write modifications into child classes directly.

Interface Segregation Principle

“Many client-specific interfaces are better than one general-purpose interface”

In the contest of classes, an interface is considered, all the methods and properties “exposed”, thus, everything that a user can interact with that belongs to that class.


image(d) Calculations_test.py file of calculator program


According to image(d), There are two abstract methods create() and convert_args_to_tuple_of_float() have been used in class Calculator. Interface is a one type of class, but it has only sign method body. So here, class calculator contains smaller interfaces(signed method body).

Now as per inheritance principle, the subclasses Addition(), Subtraction(), Multiplication(), Division() will have that interface property. subclasses will segregate this interface and each subclass will use it that interface methods.

Dependency Inversion Principle

It means that Abstractions should not depend on details. Details should depend on abstraction. High-level modules should not depend on low-level modules. Both should depend on abstractions.


image(e) Calculations_test.py file of calculator program


In calculator program, the subclasses Addition(), Subtraction(), Multiplication(), Division() are depended upon the parent class Calculator. According to the Dependency Inversion Principle the details depend on the abstractions, we can validate that with the two abstractions classes, create() and convert_args_to_tuple_of_float(). Thus. the earlier mention subclasses depend on parent class Calculator.
